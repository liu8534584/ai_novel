# InfinityWriter 前端开发指南

## 1. 技术栈初始化指令

首先，在你的项目根目录下打开终端，执行以下初始化操作：

### 初始化 Vue 3 项目 (Vite)
**指令：**
"请在当前目录初始化一个 Vue 3 项目（使用 Vite）。"

### 安装依赖
**指令：**
1. 安装 Tailwind CSS 和 Lucid-Vue-Next（图标库）。
2. 安装 Pinia（状态管理）和 Axios（请求）。
3. 设置基础的 Tailwind 配置，包含一个 dark 模式的主题色（推荐深灰色背景，类似 Obsidian 的风格）。

## 2. 核心页面：写作工作台 (Workbench.vue)

这是整个系统的灵魂。我们需要一个三栏布局：

### A. 左侧：属性监控器 (CharacterPanel.vue)
*   **功能**：实时显示 StateAgent 同步过来的主角数据。
*   **UI 元素**：HP/MP 血条、经验条、物品格（Inventory）、技能列表、当前负伤状态。

**开发指令：**
> "请编写 CharacterPanel.vue 组件。要求：
> 1. 接收一个 characterStats 对象作为 Props。
> 2. 使用 Progress Bar 显示 HP 和 MP。
> 3. 背包系统采用网格布局，鼠标悬浮显示物品详情。
> 4. 风格要像 RPG 游戏面板，使用 Tailwind 的深色卡片效果。"

### B. 中间：流式编辑器 (StreamEditor.vue)
*   **功能**：展示 AI 正在生成的文字，支持 Markdown。
*   **UI 元素**：章节标题、正文显示区、自动滚动开关。

**开发指令：**
> "请编写 StreamEditor.vue。
> 1. 实现一个打字机效果，能够接收流式（SSE）传输的文本数据并实时渲染。
> 2. 支持 Markdown 渲染（安装 md-editor-v3 或 marked）。
> 3. 在编辑器底部悬浮显示当前的字数统计。"

### C. 右侧：控制台 (ActionPanel.vue)
*   **功能**：触发自动化流程。
*   **UI 元素**：[生成大纲]、[开始自动写作]、[停止]、[保存并同步状态] 按钮，以及 AI 思考日志。

## 3. 全局状态设计 (Store/book.ts)

在全自动化过程中，前端需要管理复杂的联动状态。

**开发指令：**
> "请使用 Pinia 定义一个 useBookStore。
>
> **状态 (State) 包括：**
> *   `currentBook`: 书籍全局设定。
> *   `character`: 主角实时 JSON 数据。
> *   `chapterOutline`: 当前章节的场景列表（Scenes）。
> *   `isGenerating`: 是否正在生成中。
>
> **动作 (Actions) 包括：**
> *   `updateStats(newData)`: 当 StateAgent 返回数据时更新本地面板。
> *   `addContent(text)`: 将 AI 传回的片段追加到编辑器。"

## 4. 详细 UI 实现步骤 (Prompt 序列)

你可以按顺序使用以下指令引导 AI 完成开发：

### 第一步：布局框架
> "请帮我构建应用的主布局 App.vue。采用三栏设计：左侧 250px 宽度，右侧 300px 宽度，中间自适应。使用 Tailwind 设置背景色为 #1e1e1e（深灰），文字颜色为灰色。要求响应式，在移动端自动折叠侧边栏。"

### 第二步：模拟数据联调
> "在没有后端 API 的情况下，请在 CharacterPanel.vue 中先编写一套 Mock 数据，模拟一个末世题材主角：姓名苏哲、HP 80/100、拥有物品‘消防斧’和‘半瓶矿泉水’。请确保 UI 能完美美化这些 JSON 数据。"

### 第三步：大纲卡片交互
> "在右侧控制栏，请实现一个‘场景大纲’预览。当后端返回场景列表 JSON 时，将其显示为一张张垂直排列的卡片。每张卡片包含场景序号、地点和核心冲突。点击卡片可以手动编辑内容。"

### 第四步：流式数据接收 (SSE)
> "请在前端实现一个接收 Server-Sent Events (SSE) 的逻辑。当用户点击‘自动写作’时，建立与 /api/write 的连接，并将接收到的字符串不断追加到 Pinia 的 store 中，触发编辑器更新。"

## 🚀 为什么先开发前端能事半功倍？

1.  **可视化反馈**：当你看到左侧血条会因为 AI 写的文字而自动变短时，你会非常有动力去写后端的 Go 代码。
2.  **API 规格确认**：在写 UI 时，你会发现：“哦，我还需要一个字段来存储当前的物理负伤状态。” 这能让你提前修改 Go 的 struct 定义。
3.  **用户体验先行**：全自动写作最怕“卡死”或“无反馈”。先做前端，你能提前把 Loading 动画、AI 思考中的状态、断线重连等细节做完整。
